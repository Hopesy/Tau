# Tau.Ai Bedrock SSO baseline

Date: 2026-05-17

## Summary

Closed the P0 Bedrock AWS credential chain. After `credential_process`, `web identity`, `AssumeRole`, `ECS`, and `IMDS`, this turn adds **AWS SSO** support: when the active profile points at an SSO session (or carries legacy inline `sso_start_url`/`sso_region`/`sso_account_id`/`sso_role_name`), Tau reads the existing `aws sso login` token cache and exchanges it for short-lived role credentials via the SSO portal's `federation/credentials` endpoint.

## Changes

- Extended `BedrockProfileSnapshot` with `SsoSession`, `SsoStartUrl`, `SsoRegion`, `SsoAccountId`, `SsoRoleName`.
- Extended `BedrockProfileCredentialsResolver` to read those fields from the profile, and to look up the `[sso-session NAME]` section in the config file when `sso_session` is set (filling in `sso_start_url` / `sso_region` from there if the profile didn't repeat them).
- Added `src/Tau.Ai/Providers/Bedrock/BedrockSsoResolver.cs`:
  - returns `NotConfigured` when the profile has no `sso_account_id` / `sso_role_name`;
  - hard-fails when SSO is configured but `sso_start_url` / `sso_region` are missing;
  - locates the token cache as `~/.aws/sso/cache/<sha1>.json` where the input is the `sso_session` name (new format) or the `sso_start_url` (legacy); the directory and file are both overridable via `BedrockOptions.SsoTokenCacheDirectory` / `SsoTokenCacheFile` for tests and custom layouts;
  - parses `accessToken` + `expiresAt`, refuses expired tokens with a clear `aws sso login` hint;
  - GETs `https://portal.sso.<region>.amazonaws.com/federation/credentials?role_name=…&account_id=…` with `x-amz-sso_bearer_token`; `BedrockOptions.SsoPortalEndpoint` / `AWS_ENDPOINT_URL_SSO` override the host;
  - parses the `roleCredentials` JSON (note: `expiration` is Unix milliseconds, not ISO 8601), rejects already-expired credentials, returns `Source = "sso"`;
  - surfaces non-2xx responses with `HTTP <code>: <body>`.
- Added `SsoTokenCacheFile`, `SsoTokenCacheDirectory`, and `SsoPortalEndpoint` to `BedrockOptions`.
- Inserted the SSO resolver after `AssumeRole` and before `credential_process` in `BedrockProvider.ResolveCredentialsAsync`. Updated chain order: static → web identity → AssumeRole → SSO → credential_process → ECS → IMDS → fail.
- Added `tests/Tau.Ai.Tests/BedrockSsoResolverTests.cs` covering: `NotConfigured` when SSO fields are absent, hard error when start URL/region are missing, missing token cache, expired access token, legacy inline SSO success path (verifies SHA1(start_url) cache hash + portal host + query string + bearer header), `sso_session` success path (verifies SHA1(session name) cache hash), portal 401 passthrough.
- Updated `next.md` (P0 Bedrock chain marked `[x]`), the active port plan (current-facts checkbox flipped to `[x]`, verification list entry, two new decisions).

## Decisions

- Consumed the `aws sso login` token cache directly instead of building an OIDC registration / refresh client. Tau has no interactive SSO login UI yet (the OAuth login work is on a separate track), and `aws sso login` is the standard way users obtain SSO tokens. Reading the existing cache covers the vast majority of real workflows; token refresh and OIDC registration can be layered onto the same resolver later without changing the chain integration.
- Supported both the new `sso_session = NAME` form and the legacy inline form (`sso_start_url` directly on the profile). AWS CLI 2.9+ defaults to the session form but many existing profiles still use inline; the token cache hash input differs (SHA1 of session name vs SHA1 of start URL), so the resolver branches at that one spot and shares the rest of the flow.
- Made the token cache directory, cache file path, and portal endpoint all overridable via `BedrockOptions` (`SsoTokenCacheDirectory` / `SsoTokenCacheFile` / `SsoPortalEndpoint`) plus `AWS_ENDPOINT_URL_SSO`. This keeps tests hermetic (we never touch `~/.aws/sso/cache`) and gives operators a clean path to point at FIPS or non-AWS portals.
- Reported the canonical `expiration` from `roleCredentials` as Unix milliseconds (matching the portal's wire format) and validated against the injected clock so future caching layers can reuse the same expiration check without re-parsing.
- Surfaced portal failures verbatim (`HTTP 401: UnauthorizedException`) rather than translating to a generic "missing credentials" message so operators can tell expired-token failures apart from misconfigured account/role.
