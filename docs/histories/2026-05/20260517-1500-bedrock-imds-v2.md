# Tau.Ai Bedrock IMDS v2 baseline

Date: 2026-05-17

## Summary

Continued the P0 Bedrock AWS credential chain port. After `credential_process`, `web identity`, and `ECS container`, this turn adds **IMDS v2** (EC2 instance metadata service). When the user opts in via `AWS_EC2_METADATA_DISABLED=false` or `BedrockOptions.Ec2MetadataDisabled = false`, Tau performs the standard IMDSv2 flow — PUT `/latest/api/token` → GET `/latest/meta-data/iam/security-credentials/` → GET `/latest/meta-data/iam/security-credentials/{role}` — and feeds the resulting credentials into the SigV4 signer.

## Changes

- Added `src/Tau.Ai/Providers/Bedrock/BedrockInstanceMetadataResolver.cs`:
  - resolves the endpoint from `BedrockOptions.Ec2MetadataServiceEndpoint`, `AWS_EC2_METADATA_SERVICE_ENDPOINT`, or the default `http://169.254.169.254`;
  - enforces an IMDS-specific allowlist (169.254.169.254 / `fd00:ec2::254` / loopback / HTTPS);
  - issues IMDSv2 token PUT with `X-aws-ec2-metadata-token-ttl-seconds: 21600`, then attaches `X-aws-ec2-metadata-token` on subsequent GETs;
  - falls back to IMDSv1 (plain GETs without token) when the token request returns 4xx and `AWS_EC2_METADATA_V1_DISABLED` / `Ec2MetadataV1Disabled` is not set;
  - URL-encodes the role-name path segment, trims trailing newlines, and parses the `Code=Success` JSON response;
  - rejects non-`Success` `Code`, expired `Expiration`, missing required fields, and surfaces network / HTTP errors verbatim;
  - reads `Ec2MetadataServiceTimeout` / `AWS_METADATA_SERVICE_TIMEOUT` (default 1s) for per-request timeouts to avoid hanging on unreachable IMDS endpoints.
- Extended `BedrockOptions` with `Ec2MetadataDisabled`, `Ec2MetadataV1Disabled`, `Ec2MetadataServiceEndpoint`, and `Ec2MetadataServiceTimeout`.
- Slotted the resolver into `BedrockProvider.ResolveCredentialsAsync` after ECS (static → web identity → credential_process → ECS → IMDS → fail).
- Updated `BuildMissingCredentialsMessage` to mention the IMDS opt-in env var.
- Added `tests/Tau.Ai.Tests/BedrockInstanceMetadataResolverTests.cs` covering: JSON parsing (Success and Code-based errors), default opt-out (HTTP not called when env unset), IMDSv2 happy path (verifies the TTL header, role lookup with token, credentials GET with token, and host = 169.254.169.254), IMDSv1 fallback when token PUT returns 4xx, v1-disabled rejection, option-driven disable overriding env, endpoint override to loopback, and SSRF rejection for non-allowlisted hosts.
- Extended `tests/Tau.Ai.Tests/BedrockProviderTests.cs` with a provider-level integration regression that wires the full IMDSv2 three-step flow into the SigV4 signer (`x-amz-security-token` propagated, `Authorization` `Credential=ASIAIMDSINT/.../bedrock/aws4_request`).
- Synced `next.md`, the active port plan (current-facts checkbox, verification list, two new decisions).

## Decisions

- Defaulted IMDS to **opt-in** instead of mirroring AWS SDK's default-enabled behavior. AWS SDK assumes EC2 reachability and falls back fast on TCP refusal; Tau runs on developer laptops, containers, and restricted networks where 169.254.169.254 routinely hangs for the full timeout window. Forcing the user to explicitly enable IMDS via `AWS_EC2_METADATA_DISABLED=false` (or `BedrockOptions.Ec2MetadataDisabled = false`) avoids per-request 1s stalls; users on real EC2 instances flip the same flag to restore SDK semantics.
- Implemented IMDSv2 by default with optional IMDSv1 fallback. v2 is the only safe path against IMDS SSRF (Capital One-style attacks); v1 fallback is kept for compatibility with legacy instances and is gated by `AWS_EC2_METADATA_V1_DISABLED` / `Ec2MetadataV1Disabled` for compliance-driven deployments.
- Used a separate IMDS allowlist (169.254.169.254 + loopback + HTTPS) rather than reusing the ECS allowlist (which is scoped to 169.254.170.2/23). The two metadata services live on different fixed addresses and should not be granted to each other.
- Surfaced metadata errors verbatim (`HTTP <code>: <body>`) so that operators can distinguish "instance has no IAM role" from "IMDS endpoint unreachable" without firing up `curl` manually. Network failures show the underlying exception message to keep stack-free observability.
