# 前端协作说明

Tau 当前不是前端优先项目，**当前第一用户路径是 CLI**。  
因此这份文档的真实作用，不是规定一套已经存在的前端设计系统，而是明确：

- 当前没有成熟前端产品面
- 未来前端入口是 `Tau.WebUi`
- 在进入 Web UI 实作前，不能让前端工作反客为主

## 当前状态

- `src/Tau.WebUi/` 目前仍是占位实现。
- 当前没有需要长期维护的浏览器端设计系统、组件库或样式规范。
- 任何前端相关工作，都必须服从 `Tau.CodingAgent` 这条 P0 路径。

## 何时进入前端实现阶段

只有在下面条件基本成立后，`Tau.WebUi` 才进入真实实现阶段：

- `Tau.Ai` 与 `Tau.Agent` 边界稳定
- `Tau.CodingAgent` 的 CLI P0 路径走通
- 有最小测试与 CI 闭环
- plan 明确把 Web UI 提升为下一阶段目标

## 进入 Web UI 阶段后的默认原则

### 1. 语义先对齐，不先追求外观

Web UI 首先要对齐的是：

- 消息流语义
- 流式输出阶段
- 工具执行展示
- session / state 的真实数据来源

而不是先追求复杂视觉效果。

### 2. Web 只是宿主，不重写核心逻辑

- `Tau.Ai` / `Tau.Agent` 仍然是核心行为层
- `Tau.WebUi` 负责展示、交互和宿主装配
- 不要把原本应该在 Agent/Runtime 层的逻辑偷偷塞进 Web 层

### 3. 设计风格服从“开发工具”属性

未来 `Tau.WebUi` 默认应偏向：

- 低眩光
- 信息密度适中
- 状态清晰
- 工具输出和消息边界明确

而不是营销站式页面结构。

## 当前前端相关文档要求

在 Web UI 还未真正开始前：

- 可以记录计划、边界和约束
- 不要伪造不存在的组件规范
- 不要写看起来完整、实际没有落点的联调说明

如果未来开始实现 `Tau.WebUi`，再把以下内容补实：

- 启动与联调命令
- 页面结构与组件边界
- 浏览器验收步骤
- 前端测试策略
- 样式和设计 token 约束
