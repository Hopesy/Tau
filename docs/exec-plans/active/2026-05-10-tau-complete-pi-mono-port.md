# Tau 完整 pi-mono 移植计划

## 目标

把 Tau 从“按模块补关键切片”升级为“以 `pi-mono-main` 为参考源，逐模块追到可评审的功能等价层”。这份 plan 是完整移植的总路线图，现有 `2026-04-23-tau-port-baseline.md` 继续作为已完成基线和决策历史，本文件负责后续 parity closure。

“完整”在这里不是一次性把所有代码机械翻译到 C#，而是每个上游能力都要在 Tau 中落成下列三件事：

- 有明确的 Tau-native 边界和数据模型。
- 有本地可验证的 test 或 smoke。
- 文档、history、quality、next 同步说明已完成/未完成，不把 seam 伪装成完整功能。

## 当前执行切片

当前执行切片已推进到 `Tau.CodingAgent` 的 JSONL session tree、HTML transcript export、HTML text code fence rendering、HTML inline code span rendering、HTML plaintext link rendering、HTML Markdown heading/list/blockquote block rendering、HTML Markdown strong/emphasis span rendering、HTML Markdown pipe table rendering、HTML Markdown task list rendering、HTML image metadata caption、HTML long tool result folding、HTML tool call JSON argument rendering、HTML long tool call arguments folding、prompt/skill、JSON extension command/resource/diagnostics、失败普通回合 rollback、host-level retryable error auto-retry、retry audit event、Tau-native retry settings command 和 context overflow compact-and-retry baseline：

- 新增 `CodingAgentTreeSessionStore` / `CodingAgentTreeSessionController`，保留 flat JSON snapshot 兼容入口，同时默认维护 `.tau/coding-agent-session.jsonl`；`TAU_CODING_AGENT_TREE_SESSION_FILE` 或 `.jsonl` 形式的 `TAU_CODING_AGENT_SESSION_FILE` 可指定 tree session 文件。
- JSONL 第一行写 `type=session` header，后续 append-only 写 `message`、`model_change`、`session_info`、`label` entries，每个 entry 带 `id`、`parentId`、`timestamp`；host 通过 runner message count diff 追加新消息，避免重写旧 history。
- `/session` 同时报告 flat stats、估算 token/context usage、auto-compaction threshold budget、当前 retry policy 和 tree file/leaf/entry/message/branch/label/cwd/parent 信息；`/tree [max entries] [default|no-tools|user-only|labeled-only|all] [--label-time] [--search query]` 输出短 id、parent、entry 摘要、label、label timestamp、当前 branch 与 leaf 标记，并在 header 显示 cwd / parent metadata，按上游 tree selector 的核心过滤语义隐藏 bookkeeping / tool-only assistant / tool result；`--search` 会按空格分词匹配 entry text、label 和基础 metadata；未显式传 filter 时读取 settings `treeFilterMode`，无效值回退 `default`；`/label <entry-id> [label | clear]` 追加 label change；`/fork <entry-id>` 从历史 entry 切出新 branch 并恢复当前 runner messages；`/clone` 把当前 active branch 导出成新的 JSONL session 并切换当前 runner/session controller；`/resume [latest | path.jsonl]` 恢复 JSONL session。
- `/export` 默认导出 standalone HTML transcript；`/export <path.html|path.htm>` 显式导出 HTML；HTML 提供 branch outline、`default/no-tools/user-only/labeled-only/all` filter/search、cwd/parent metadata、内嵌当前 branch JSONL 并提供 Download JSONL 按钮，按 current branch JSONL 渲染 message / session_info / model_change / label / compaction timeline，消息文本里的 fenced code block 会拆成独立 `code-block` / `<code>` 区块，普通文本里的 backtick inline code span 会渲染为 `<code class="inline-code">`，普通文本段会把 Markdown-style `[label](http/https...)` 和裸 `http(s)` URL 渲染成外链，code fence 和 inline code 内不链接，非 http(s) scheme 保持文本安全输出；普通文本段检测到 heading/list/blockquote block marker 时会改用轻量 rich-text block rendering，渲染 `h1`-`h6`、`ul/ol/li` 和 `blockquote`，并继续复用 inline code/link 安全输出；普通文本里的 `**strong**` / `__strong__` 和 `*emphasis*` / `_emphasis_` 会分别渲染为 `<strong>` / `<em>`，并复用 inline code/link 安全输出；inline code、code fence 和单词内部下划线不会触发 emphasis；普通文本段中的 Markdown pipe table（header + separator + data rows）会渲染为可横向滚动的 `<table>`，单元格继续复用 inline code/link/strong/emphasis 安全输出，code fence 内表格文本保持代码块；普通文本列表项里的 `[ ]` / `[x]` task marker 会渲染为 disabled checkbox，任务文本继续复用 inline code/link/strong/emphasis 安全输出，code fence 内任务列表文本保持代码块；图片内容继续以内嵌 data URI 渲染，并在 caption 显示 mime type 和估算字节数；长 tool result 文本会默认折叠为 `<details class="tool-result-fold">` 并保留完整输出；tool call arguments 中可解析 JSON 会格式化为 `code-block` / `<code data-language="json">`，不可解析参数仍按原始 `<pre>` 安全转义，超长参数会默认折叠为 `<details class="tool-call-arguments-fold">` 并保留全文；消息节点绑定 JSONL entry id 和 label badge，支持 copy link 和 `targetId` deep link 自动定位；`/export <path.jsonl>` 导出当前 branch 为独立 JSONL session；`/share` 会复用同一 HTML transcript exporter，检查 GitHub CLI 登录状态并创建 secret gist，预览 URL 支持 `TAU_SHARE_VIEWER_URL` 覆盖；`/import <path.jsonl>` 等价于 resume；其他非 `.jsonl/.html/.htm` 的 `/export` / `/import` 继续走 Tau flat snapshot JSON，避免破坏旧 session。
- prompt template discovery/expansion baseline 已接入：用户目录 `~/.tau/prompts`、项目目录 `./.tau/prompts` 和 `TAU_CODING_AGENT_PROMPT_PATHS` 指定路径可提供 `.md` prompt template；`/prompts` 列表展示模板，非内置 slash 输入命中模板时会在 runner 调用前展开参数占位。
- skill command discovery/expansion baseline 已接入：用户目录 `~/.tau/skills`、项目目录 `./.tau/skills` 和 `TAU_CODING_AGENT_SKILL_PATHS` 指定路径可提供 `SKILL.md`；`/skills` 列表展示 `/skill:<name>` 命令，`/skill:<name> args` 会把 skill body 包装成上游风格 `<skill>` block 后发送给 runner；可见 skill 会进入默认 system prompt inventory，`disable-model-invocation: true` 只保留显式命令调用。
- JSON extension command/resource/diagnostics baseline 已接入：用户目录 `~/.tau/extensions`、项目目录 `./.tau/extensions` 和 `TAU_CODING_AGENT_EXTENSION_PATHS` 指定路径可提供 `.json` command definitions；`/extensions` 列表展示命令、extension JSON 文件路径、每个文件的 command/prompt/skill resource 计数、重复 command 的解析名、prompt/skill resource paths，以及坏 JSON、不可读文件、缺失显式路径等 load diagnostics；非内置 slash 输入先查 extension command，再查 skill/prompt；status-only command 直接写 UI，`sendToRunner=true` command 会展开 `prompt/response` 后发送给 runner，重复 name 按上游规则解析成 `name:1`、`name:2`；同一 JSON 可通过 `promptPaths/skillPaths` 或 `resources.promptPaths/resources.skillPaths` 贡献 prompt/skill 发现路径，路径相对 extension JSON 所在目录解析。
- Compaction recent-message retention baseline 已接入：manual / auto compaction 写 `firstKeptEntryId` 时默认先按 `TAU_CODING_AGENT_COMPACT_KEEP_RECENT_TOKENS` 的 token budget 从尾部选择 cut-point，未命中时再保留当前 compaction boundary 后最近 4 条 message，可通过 `TAU_CODING_AGENT_COMPACT_KEEP_RECENT_MESSAGES` 调整 fallback；如果 cut point 落在一个 user turn 中间，Tau 会在 compaction entry 上写 `isSplitTurn=true` 和确定性 `turnPrefixSummary`，branch restore / resume / clone export 按 summary + split-turn prefix context + retained messages + post-compaction messages 重建 runtime，并对导出的 JSONL session remap `firstKeptEntryId`。
- Context overflow compact-and-retry baseline 已接入：host 遇到 context overflow 错误型 `AgentEndEvent` 或 runner exception 时，会恢复失败回合前 snapshot，调用 `CompactAsync()`，把结果记录为 `fromHook=true` compaction entry，更新回滚基线后重试同一输入；如果 compaction 失败，则回滚到原始 snapshot，不写失败输入。
- 失败普通回合 rollback baseline 已接入：host 在普通消息调用 runner 前记录回合前 snapshot，runner exception、取消或错误型 `AgentEndEvent` 后恢复 snapshot，flat JSON 和 JSONL tree 只持久化回滚后的 messages/provider/model/name。
- Host-level retryable error auto-retry baseline 已接入：生产入口优先从 settings retry fields 读取配置，未配置时回退 `TAU_CODING_AGENT_AUTO_RETRY_ATTEMPTS` 和 `TAU_CODING_AGENT_AUTO_RETRY_BASE_DELAY_MS`，retryable runner exception 或错误型 `AgentEndEvent` 会先恢复回合前 snapshot，再按 exponential delay 重试同一输入；成功后只持久化成功 attempt，非 retryable 错误或重试耗尽继续按 rollback baseline 收口。
- Tau-native retry audit event baseline 已接入：JSONL tree 会为普通 transient retry 写 `auto_retry_start` 和 `auto_retry_end` entries，记录 attempt/maxAttempts/delay/error/success/finalError；成功恢复时先同步成功 attempt 的 user/assistant messages，再追加 retry end；失败或耗尽时保留 retry audit，但不写失败输入或 partial assistant。`/tree` search 和 HTML transcript timeline/branch outline 已能显示/search retry entries。
- Tau-native retry settings command baseline 已接入：`/retry [current|default|off|<max attempts> [base delay ms]]` 可查看、关闭、恢复 env/default 或设置 attempts/base delay，写回现有 settings JSON，并通过 host callback 让同进程下一轮普通输入立即使用新策略；retry delay 被 cancellation token 中断时会显示明确 `auto-retry cancelled during delay` 状态，写 `auto_retry_end(success=false, finalError="Retry cancelled")`，并回滚失败输入；这仍不是完整上游 RPC/settings UI/cancellation parity。
- 当前仍不把真正的 interactive tree navigator、Tau 专属 share viewer、完整 Markdown/highlight renderer、完整上游 richer HTML template/vendor renderer、上游 LLM-generated split-turn summarization、auto-retry RPC/settings UI parity、完整 retry cancellation UI、完整 compaction extension events/cancellation UI、完整 TypeScript extension runtime、custom tools/events、theme loader、interactive resource selector 和 richer runtime diagnostics 写成已完成。

- 2026-05-17：决定让 ECS resolver 接在 `credential_process` 之后、`fail` 之前，而不是放在 static/profile 之间。原因是 ECS 触发条件是 env vars（典型容器内部署），credential_process 是显式 profile 配置（用户主动开启）。优先让用户显式配置的 helper 起作用，再回退到容器自动注入的 task role 凭证，与“显式优先”的整体取舍一致；同时把 IMDS 留到这个位置之后，避免在显式 helper 失败时静默退到 metadata service。
- 2026-05-17：决定把 Bedrock IMDS 默认改为 **opt-in**（`AWS_EC2_METADATA_DISABLED=false` 或 `Ec2MetadataDisabled=false` 才启用），与 AWS SDK 默认 opt-out 行为相反。原因是 Tau 作为编码 agent 经常运行在开发机/本地容器/受限网络上，169.254.169.254 通常会触发 TCP 卡顿（不像 ECS endpoint 只在容器里出现）。把默认改为 opt-in 避免每条 Bedrock 请求都为不存在的 metadata service 等 1s+；EC2 实例上用户显式 `AWS_EC2_METADATA_DISABLED=false` 即可拿回 SDK 默认行为。
- 2026-05-17：决定让 IMDS 默认走 IMDSv2（PUT token + 后续 GET 携带 `X-aws-ec2-metadata-token`），仅在 token 请求 4xx 且 `AWS_EC2_METADATA_V1_DISABLED` 未启用时回退 v1。原因是 v2 是 AWS 推荐姿态，能避免 IMDS SSRF 旁路；v1 仅作为旧实例兼容路径，并允许通过 env / option 强制关闭以匹配安全合规需求。
- 2026-05-17：决定先把 AssumeRole 切到 `source_profile`-only（指向静态 profile），credential_source 与递归 source_profile 留后续。原因是大多数用户的 ~/.aws/config 是“base 静态 + dev role_arn + source_profile = base”形态；先打通该路径已经能消费泛化后的 SigV4 签名器和共享 STS XML 解析器，并避免一次性引入递归依赖、credential_source 的多分支组合（Environment / EcsContainer / Ec2InstanceMetadata）和潜在循环。
- 2026-05-17：决定补完 `credential_source` (Environment / EcsContainer / Ec2InstanceMetadata) 三种 AssumeRole 源，重用现有 ECS/IMDS resolver 的 HTTP 路径，并对 `credential_source=Ec2InstanceMetadata` 强制开启 IMDS（覆盖默认 opt-out）。原因是 profile 显式声明 `credential_source = Ec2InstanceMetadata` 已经表明用户在 EC2 上使用 IMDS 作为 AssumeRole 源；若仍受全局 opt-out 控制会产生静默失败。同时仍未支持递归 source_profile，因为递归依赖管理与防循环值得单独切片。
- 2026-05-17：决定让 `ResolveStaticCredentials` 在 active profile 含 `role_arn` 时主动放弃 profile-static，与 AWS SDK 行为对齐。原因是 profile 同时配置 `role_arn` + 静态凭证时，静态凭证是 AssumeRole 的源凭证，不应该被当作 profile 的最终凭证返回，否则 chain 会在 AssumeRole 之前就用错凭证直接签 Bedrock 请求。
- 2026-05-17：决定把 SigV4 签名器泛化到 `(region, service)` 而不是再克隆一份 STS 专用签名器。原因是 STS / Bedrock / 后续可能的 SSO / IAM endpoint 都用同一套 v4 算法；service 进入 credential scope 与 signing key 派生即可，签名器其余逻辑没有任何分支差异。同时 STS XML 解析（成功/错误两条路径）抽到 `BedrockStsResponseParser` 共享，避免 web identity 与 AssumeRole 维护两套不同的解析器。
- 2026-05-17：决定让 Bedrock SSO 直接消费 `aws sso login` 写下的 `~/.aws/sso/cache/<hash>.json` token cache，而不是自己实现 SSO-OIDC token registration / refresh。原因是 Tau 当前没有交互式登录入口（OAuth login 是另一条线），而 `aws sso login` 已经是 AWS CLI 用户的标配。先把已有 token 转成 Bedrock 凭证就能解锁 99% 的 SSO 用例；token 自动刷新和真实 OIDC 注册留到后续切片，按需扩展同一 resolver。
- 2026-05-17：决定让 SSO resolver 同时识别 `sso_session = NAME` 与 legacy inline (`sso_start_url` 直接写在 profile 里) 两种形态，并按 AWS CLI 规则选择哈希输入（SHA1(session name) vs SHA1(start_url)）。原因是上游 AWS CLI 自 2.9 起推荐 `sso-session` 形态，但仍有大量旧 profile 写 inline；同时把 token cache 目录抽成 `BedrockOptions.SsoTokenCacheDirectory` / `SsoTokenCacheFile` 让单元测试不依赖 `~/.aws/sso/cache`。
